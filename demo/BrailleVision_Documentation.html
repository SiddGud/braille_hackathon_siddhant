BrailleVision: Technical Documentation

Developed By: Solo Developer Submission

Project Goal: A real-time, hands-free web application powered by a hybrid pipeline of deep learning and classical computer vision to translate physical Braille to English text and speech.

1. The 4-Stage Image Processing Pipeline

Unlike traditional OCR systems, BrailleVision is built specifically to handle the shadows, gradients, and noisy textures of physical embossed paper using a custom 4-stage pipeline:

  • Stage 1 (Preprocessing): Incoming images are passed through an OpenCV Bilateral Filter (cv2.bilateralFilter). This mathematical filter smooths out paper grain and lighting artifacts while preserving the sharp, high-frequency edges of the physical Braille dots.
  • Stage 2 (Test-Time Augmentation): The cleaned image is fed into a fine-tuned YOLOv8-m foundation model. Crucially, inference is run with TTA (Test-Time Augmentation) enabled, forcing the model to evaluate the image at multiple zoom scales simultaneously to catch faint or hidden dots.
  • Stage 3 (Inference & Decoding): The YOLO model predicts bounding boxes around Braille cells and classifies them into 6-bit binary arrays, mapping the raised dot patterns directly to the English alphabet.
  • Stage 4 (Post-Processing & Heuristics): Bounding boxes are spatially sorted to establish lines and word boundaries by measuring horizontal gaps between cells. Finally, because physical paper suffers from spatial compression, the raw string is passed through a custom Python difflib layer that acts as a context-aware spellchecker to correct minor character hallucinations.

2. Engineering Journey & Challenges

Challenge 1: The "Heavy Model" Hallucination

Even though the initial massive 11M+ parameter models hit 93% accuracy during training on cloud GPUs, they hallucinated wildly on physical paper due to shadows. The solution was not a larger model, but rather pivoting to a hybrid architecture: pairing a lighter foundation model with intense OpenCV classical preprocessing.

Challenge 2: The Flip Augmentation Disaster

Early on, the model kept confusing the letter "r" and "w". The training script was automatically flipping images horizontally (fliplr=0.5) to artificially increase dataset size. Because Braille is chiral, mirroring an "r" literally creates a "w". Disabling all flip augmentations permanently resolved the systemic character confusion.

3. Architecture Stack

  • Frontend: Custom-built with HTML5, CSS3, Vanilla JavaScript, and the Canvas API. Optimized for extreme accessibility, featuring full screen-reader compatibility and hands-free Web Speech API controls.
  • Backend API: Python 3.11, FastAPI, and Uvicorn. Exposes low-latency RESTful endpoints for real-time video translation.
  • Machine Learning Layer: PyTorch, Ultralytics YOLOv8-m, OpenCV (Bilateral Filtering), and Google Text-to-Speech (gTTS).

4. Dataset Curation

The foundation model was trained on a meticulously curated dataset of ~1,800 images, combining the Angelina Braille Dataset, the DSBI double-sided dataset, and custom Kaggle imagery. Physical test images were manually captured to tune the mathematical preprocessing and autocorrect heuristic layers.
